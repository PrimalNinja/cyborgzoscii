Overview

The ZOSCII Tamperproof Blockchain (ZTB) provides:

  • ✅ Quantum Proof - Security based on combinatorial impossibility, not computational difficulty
  • ✅ Scalable Architecture - Trunk and branch chains for efficient organization
  • ✅ Tamperproof Guarantee - ~10^152900 impossibility of forgery
  • ✅ Fast Verification - O(1) lookups, no complex cryptography
  • ✅ Information-Theoretic Security - Immune to quantum algorithms

Core Components

1. Genesis ROM (64KB)

Immutable root of trust. Contains random data with all 256 byte values (0-255) present.

⚠️ Lost Genesis ROM = Unrecoverable Blockchain

2. Rolling ROM (64KB)

Built from samples of previous blocks (up to 64 blocks × 1KB each). Acts as encoding table for each new block.

3. ZOSCII Encoding

Each byte (1 byte) → pointer (2 bytes) into Rolling ROM. Random selection creates non-deterministic encoding.

Architecture

        ↓
    Trunk Block 1 (encoded)
        ↓
    Trunk Block 2 (encoded)
        ├─→ Branch A Block 1 (encoded)
        │       ↓
        │   Branch A Block 2 (encoded)
        │
        ├─→ Branch B Block 1 (encoded)
        │       ↓

Block Types

  • Trunk Block (is_branch = 0) - Main chain blocks, linear progression
  • Branch Block (is_branch = 1) - Splits from trunk, maintains parent link

Command-Line Tools

1. ztbcreate - Genesis ROM Creator

Creates the 64KB high-entropy Genesis ROM file.

ztbcreate <entropyfile1> [<entropyfile2>] [<entropyfile3>] <output_genesis_rom>

Example:

ztbcreate selfie.jpg genesis.rom
⚠️ CRITICAL: Keep this file secure and backed up. Loss = unrecoverable blockchain.

2. ztbaddblock - Block Creator

Adds a new block to an existing chain (trunk or branch).

# Add text block
ztbaddblock <genesis_rom> <chain_id> -t "text string" [-x1] [-x2] [-i <block_id>]

# Add file block
ztbaddblock <genesis_rom> <chain_id> -f <file_path> [-x1] [-x2] [-i <block_id>]

Examples:

# First block creates new trunk
ztbaddblock genesis.rom MyTrunk -t "Genesis block"

# Add subsequent blocks with custom ID
ztbaddblock genesis.rom MyTrunk -t "Block 2" -i 1
ztbaddblock genesis.rom MyTrunk -f invoice.pdf -i "invoice-001"

Options:

  • -i "id" - Supply custom block ID (max 36 chars, truncated if longer)
  • -t "text" - Add text string as payload
  • -f filename - Add file contents as payload
  • -x1 - X1 extended security: prev-block CRC binding, no on-disk XOR
  • -x2 - X2 extended security: prev-block CRC binding + on-disk XOR

3. ztbaddbranch - Branch Creator

Creates a new branch splitting from a trunk block.

ztbaddbranch <genesis_rom> <trunk_id> <new_branch_id> -t "text" [-x1] [-x2] [-i <block_id>]
ztbaddbranch <genesis_rom> <trunk_id> <new_branch_id> -f <file> [-x1] [-x2] [-i <block_id>]

Example:

# Create trunk
ztbaddblock genesis.rom CompanyLedger -t "2025 Books"

# Branch off for departments
ztbaddbranch genesis.rom CompanyLedger Sales -t "Sales Genesis"
ztbaddbranch genesis.rom CompanyLedger Marketing -t "Marketing Genesis" -i "marketing-root"

4. ztbfetch - Block Fetcher & Decoder

Retrieves and decodes a specific block, verifying its integrity.

ztbfetch <genesis_rom> <chain_id> <block_index>

Examples:

# Fetch from trunk
ztbfetch genesis.rom MyTrunk 3

# Fetch from branch
ztbfetch genesis.rom ClientA 2

5. ztbverify - Chain Verifier

Verifies integrity of chains by working backwards from latest block to Genesis.

# Verify trunk and all branches (default)
ztbverify <genesis_rom> <trunk_id>

# Verify trunk only
ztbverify <genesis_rom> <trunk_id> -t

# Verify all branches only
ztbverify <genesis_rom> <trunk_id> -bb

# Verify specific branch
ztbverify <genesis_rom> <trunk_id> -b <branch_id>

6. ztbcheckpoint - Checkpoint Creator

Creates a new Genesis ROM from the current rolling ROM state, enabling a clean chain split for archival.

ztbcheckpoint <genesis_rom> <chain_id> <output_checkpoint_rom>

Example:

# At end of financial year
ztbcheckpoint genesis.rom FY2024 checkpoint.rom

File Formats

Genesis ROM File

  • Size: 65,536 bytes (64KB)
  • Content: High-entropy random data
  • Format: Raw binary (no structure)
  • Requirements: All 256 byte values (0-255) must be present

Block Filename Convention

<chain_id>_<index>_<block_id>.ztb

Examples:

MyTrunk_0001_A0B9845E-1234-4567-89AB-CDEF01234567.ztb
ClientA_0002_1.ztb
Sales_0010_invoice-001.ztb

Rolling ROM Construction

Construction Algorithm

  1. Start with empty 64KB buffer
  2. Sample from BRANCH BLOCKS (if on branch):
    • Copy first 1KB of ENCODED data from each block
    • Maximum: 64 blocks
  3. Sample from TRUNK BLOCKS (if space remains):
    • Copy first 1KB of ENCODED data
    • Continue until buffer full
  4. Fill remainder with GENESIS ROM data

Why ENCODED Data?

The encoded addresses themselves are the entropy source. Original payload might be boring, but random address selection creates high entropy. Copying encoded addresses preserves this entropy.

Security Model

Tamperproof Guarantee

If an attacker modifies Block N:

  1. Block N's encoded data changes
  2. Block N+1's pointers now point to wrong locations
  3. Probability of pointers still resolving correctly: ~10^-152900

This is combinatorially infeasible - not just hard, but information-theoretically impossible.

X1 and X2 Extended Security Modes

Both modes add a dual-CRC binding layer on top of standard tamper-evidence. Every X1/X2 block commits to two CRC32 values in its prefix:

  1. Current block CRC — over the ZOSCII-encoded block data.
  2. Previous block on-disk CRC — over the complete previous block file (fully encoded, including prefix), committed to by the current block and verified by the next.

The difference between the two modes:

  • -x1 — dual-CRC binding only. No on-disk XOR.
  • -x2 — dual-CRC binding plus the output block is physically XOR'd against the previous block file on disk.

1KB threshold: The rolling ROM samples the first 1KB of each block. For payloads under 1KB the entire block is captured in the rolling ROM — the combinatorial tamper-evidence applies to the full payload without either mode. For payloads exceeding 1KB, content beyond that boundary is protected only by CRC32 in normal mode. X1 and X2 both close this gap.

Trunk block 1 exception: The first block of a trunk has no predecessor and is stored plain even in X1/X2 mode. Its prev-block CRC is zero. This is by design and handled correctly by all tools.

Example Workflows

Basic Trunk Chain

# 1. Create Genesis ROM
./ztbcreate selfie.jpg genesis.rom

# 2. Create trunk chain
./ztbaddblock genesis.rom MyTrunk -t "Block 1"
./ztbaddblock genesis.rom MyTrunk -t "Block 2" -i 2
./ztbaddblock genesis.rom MyTrunk -t "Block 3"

# 3. Verify
./ztbverify genesis.rom MyTrunk

# 4. Fetch specific block
./ztbfetch genesis.rom MyTrunk 2

Hierarchical Organization

# 1. Create genesis
./ztbcreate selfie.jpg genesis.rom

# 2. Create company trunk
./ztbaddblock genesis.rom CompanyLedger -t "FY2025 Genesis"

# 3. Branch for departments
./ztbaddbranch genesis.rom CompanyLedger Sales -t "Sales Dept"
./ztbaddbranch genesis.rom CompanyLedger Marketing -t "Marketing Dept"

# 4. Add department records
./ztbaddblock genesis.rom Sales -f invoice_001.pdf -i "inv-001"
./ztbaddblock genesis.rom Marketing -f campaign_q1.doc

# 5. Verify everything
./ztbverify genesis.rom CompanyLedger

Important Notes

Critical Requirements

  • Genesis ROM is Irreplaceable - Back up to multiple locations
  • Entropy Matters - Must contain all 256 byte values
  • No Backwards Compatibility - Version control carefully
  • Storage: 2× Size - Encoded blocks are double the raw size
  • Block ID Limit: Max 36 characters (truncated if longer)

Best Practices

  • Regular verification (daily/weekly)
  • Genesis ROM: 3+ copies, offline storage
  • Descriptive chain IDs and block IDs
  • Monitor verification failures
  • Secure Genesis ROM with encryption
