Abstract

The ZOSCII Tamperproof Blockchain introduces a revolutionary security model for Distributed Ledger Technology (DLT), decoupling structural integrity from data confidentiality. By leveraging Information-Theoretically Secure (ITS) encoding and Combinatorial Security, this blockchain is inherently Quantum Proof and does not rely on traditional, computationally intensive ciphers (like SHA-256) or mathematically difficult problems (like factoring). Its unique pointer-based block linking, anchored by a high-entropy GENESIS BLOCK, makes the entire ledger history tamperproof through combinatorial impossibility, while a branch-chain architecture enables unprecedented scalability and indexing efficiency.

1. Introduction: The Quantum and Scalability Challenges

Traditional blockchain security is predicated on computational difficulty—the idea that the work required to forge a block (finding a valid hash) is computationally impractical. This foundation faces two existential threats:

Quantum Threat

Algorithms like Shor's could compromise the underlying Public Key Cryptography (used for key exchange and digital signatures), while Grover's algorithm could significantly diminish the effective security of hashing functions, potentially breaking current Proof-of-Work systems.

Scalability

Linear chain architectures require all nodes to process and store all data, leading to slow synchronization and burdensome wallet lookups.

Solution: The ZOSCII Tamperproof Blockchain solves these by shifting the security burden from computationally difficult proofs to mathematically guaranteed combinatorial impossibility and introducing a scalable architecture optimized for verification.

2. Core Security Mechanism: ITS and Combinatorial Impossibility

The security of the ZOSCII Tamperproof Blockchain relies on three interlocking components:

A. The Genesis Anchor (GENESIS BLOCK)

The chain begins with a Genesis Block: a 65,536-byte file whose first byte marks it as the genesis type, with the remaining 65,535 bytes being high-entropy data. This file acts as the immutable, unpredictable root of trust for the entire system. All subsequent block consensus checks must ultimately trace their integrity back to this starting point — either directly, or via a Truncation block that has assumed its role for an archived chain.

Critical Requirement: The genesis entropy region should contain a good spread of all 256 byte values. Loss of this file (before a checkpoint and truncation establish a new self-contained root) makes the entire blockchain unrecoverable.

B. The Rolling Chain of Custody (ROLLING ROM)

Subsequent blocks do not link via a single cryptographic hash alone. Instead, each new block is encoded using a 64KB ROLLING ROM, which is a composite sample of up to 1KB of each block's complete serialized bytes, drawn from each of up to 64 previous blocks in its lineage, walking backwards via each block's recorded previous-block ID.

This process creates a dynamic, explicit, and verifiable genetic code for the new block, directly linking it to its entire lineage.

Rolling ROM Construction Priority:
  • First: walk backwards via prev_block_id, sampling 1KB from each block in order (most recent contributes last in the walk, oldest first into the buffer)
  • Second: if a Truncation block is reached, its stored 64KB ROM snapshot fills any remaining space
  • Third: otherwise, fill any remaining space from the Genesis Block

Maximum 64 blocks contribute (64KB ÷ 1KB per block).

C. The Non-Reproducible Pointer Encoding

The payload data within a block is encoded using a unique ZOSCII method that utilizes the block's own ROLLING ROM (built from its predecessors) as an address table:

Encoding: A 1-byte data value (from the payload) is encoded as a 2-byte address pointer. The system takes the 1-byte value and finds a random location within the 64KB ROLLING ROM that contains that desired value. The 2-byte address of that random location becomes the encoded value in the new block.

Decoding: Decoding is extremely fast, requiring only a simple lookup of the 2-byte address pointer in the ROLLING ROM to retrieve the original 1-byte value.

The Tamperproof Guarantee:

If an attacker alters the serialized bytes of Block N, every block whose ROLLING ROM sampled Block N would need its 2-byte pointers to still coincidentally resolve to the same original values against the altered bytes.

Number of valid permutations: ~10^152,900

This is combinatorially infeasible — not just hard, but information-theoretically impossible, even for quantum computers.

The security is based on the non-reproducible randomness of the address selection during encoding.

D. Dual-Hash Block Binding

Always-Active Integrity:

Every block commits to two integrity hashes within its ZOSCII-encoded section: a hash over the padded payload only — not the raw header, not the link/length metadata — computed before ZOSCII encoding is applied; and a hash of the previous block's complete serialized bytes (the whole on-disk file, header included). Both use the same algorithm, determined by the hash_type byte stored alongside them. This dual-hash binding is a standard feature of all blocks — not an optional mode. The asymmetry is deliberate: the current block's payload may eventually be hashed incrementally as it streams in, so it must never be hashed together with the header, which is always small and fully materialized in memory; the previous block, by the time it is hashed, is already a finished and fully-committed artifact, so hashing it whole carries no such constraint. The prevHash field is zero for the first block of a chain (no predecessor).

This creates a dual-hash constraint: an attacker attempting to forge block N must simultaneously satisfy two interdependent integrity checks. The second hash target is over the previous block's serialized form, which is itself a product of non-deterministic ZOSCII encoding. Because the attacker cannot predict or control the encoded output, satisfying the second constraint is practically infeasible independently of the first. The current-block hash being payload-only does not weaken protection of the header fields (prev_block_id, trunk_id, is_branch, block_id): those are instead protected by the rolling ROM, since the header sits within the first 1KB of each block's on-disk bytes that gets sampled into every dependent block's rolling ROM, subjecting header tampering to the same combinatorial tamper-evidence described in Section C.

E. Hash Algorithm Selection

The hash algorithm and scope are stored alongside the hash values and read back by all tools, enabling future algorithm agility without breaking the format. Four options are defined:

The ZRollingHash variants (2 and 3) are implemented in the C# reference library. The C command-line tools currently recognize these values but compute them as a stub (returning zero); only CRC32Full and CRC321KB are functionally implemented in C at this time. CRC32Full is the algorithm implemented by the C tooling and used for cross-compatibility with it; the C# reference library defaults to RollingFull when no hash type is specified.

1KB Windows — Related but Distinct:

The rolling ROM samples exactly 1KB from each previous block's complete serialized bytes — the 111-byte header plus the start of the encoded section. CRC321KB and Rolling1KB hash a different 1KB window: the first 1024 bytes of the padded payload alone, with no header mixed in. The two windows are related in spirit — both exist to bound the cost of full-block processing — but they are not byte-identical, since one always includes the header and the other never does.

  • Payload tampering, any hash type: caught by the current-block hash directly (in full under the Full variants, within the first 1KB under the windowed variants), and reinforced by the rolling ROM's own combinatorial tamper-evidence for whatever portion it samples.
  • Header/metadata tampering (prev_block_id, trunk_id, is_branch, block_id): not covered by the current-block hash under any hash type, since that hash is payload-only. Protected instead by the rolling ROM: the header is always within the first 1KB of a block's on-disk bytes, so it is always sampled into dependent blocks' rolling ROMs, making it subject to the same combinatorial guarantee as payload content within that window.

Rolling1KB is particularly well-aligned architecturally: it applies the same ZRollingHash algorithm used in the ZOSCII encoding itself, keeping the security model internally consistent.

F. Hashing Is Detection, Not Security

It is worth being precise about what the hash fields do and do not provide. The security guarantee — the ~10^152900 combinatorial impossibility described in Section C — comes entirely from ZOSCII's pointer-based encoding against the rolling ROM, and does not depend on hashing at all. A correctly-chosen tampered byte cannot be predicted to decode to any particular value without already knowing the rolling ROM it was encoded against; that property holds with the hash fields removed from the format entirely.

What the hash fields add is cheap, early detection. Without them, a tamper outside the region any rolling ROM actually samples (beyond the first 1KB of a block's on-disk bytes) would decode silently to altered content, undetected, unless and until some other check happened to notice. The hash gives an immediate, inexpensive way to catch tampering within its window — a single comparison, rather than an attempted decode — at the cost of only protecting the bytes it actually examines.

Why the Stored Hash Cannot Simply Be Rewritten:

The hash value itself lives inside the first 1KB of a block's on-disk bytes — the same region the rolling ROM samples into every dependent block. This makes the stored hash readable but not freely rewritable: an attacker cannot quietly overwrite the hash field to match a tampered payload, because any edit within that region is itself subject to the combinatorial tamper-evidence of Section C. The attacker's only avenue is the reverse — leave the stored hash untouched, and search for a tampered payload, confined to the region the hash does not visibly protect, that happens to still produce the same fixed hash value.

Because every block also commits to a hash of the block before it (Section D), a tamper to block N is checked against two independent, fixed targets rather than one: N's own current-block hash (if the tamper falls within N's payload-hash window) and N+1's prevHash (if the tamper falls within the on-disk window N+1 committed to). A successful forgery of N must satisfy both targets that apply to it, computed over two different, only partially overlapping windows of N's bytes.

G. Why ZRollingHash, Not CRC32, for This Role

CRC32 is built on a fixed, public, universal structure: a single polynomial constant (0xEDB88320) applied identically to every input, for every block, in every chain, forever. An attacker does not need access to any specific block's data to study how CRC32 responds to edits — the function's behavior can be characterized once, in isolation, and that knowledge applies everywhere. CRC32's linearity over GF(2) means that, given a target checksum, it is a known and practical technique to compute compensating byte edits that leave the checksum unchanged — CRC32 was designed to catch accidental, random bit errors, not to resist an adversary deliberately searching for collisions.

ZRollingHash has no equivalent external structure. Its four-pass construction folds the input against itself — pass p XORs together every byte at stride-4 positions p, p+4, p+8, ... — with no polynomial, no table, no seed, and no constant of any kind. The hash's sensitivity to any given edit is a property of that block's own bytes alone; there is nothing fixed for an attacker to pre-study independent of the specific data being attacked.

The Stride Misalignment:

ZOSCII encoding groups bytes in pairs: each payload byte becomes a 2-byte address. ZRollingHash groups bytes in fours, by position modulo 4, with no fixed relationship to where any address pair begins or ends. A single tampered on-disk byte — really, one half of one 2-byte address — lands in exactly one of the four stride classes, perturbing only that pass's running XOR fold. To leave the stored hash unchanged, an attacker needs a second edit, in the same stride class, that folds the first edit's effect back out — while that second edit must also remain a valid ZOSCII address, must not disturb whichever of N's own hash window or N+1's prevHash window it falls into, and must not bleed into any of the other three stride classes.

Where CRC32's linear algebra would let an attacker solve directly for such a compensating edit, ZRollingHash offers no such shortcut. The attacker is left with an unstructured search across four simultaneous fold constraints, not one solvable equation.

H. Can a Forgery Be Meaningful?

Suppose, hypothetically, an attacker found a tampered byte sequence that satisfied both applicable hash targets. This is not the same as having produced useful forged content. Surviving a hash check only means the forged bytes happened to fold to a specific fixed 4-byte (or 4-byte CRC32) target — it says nothing about what those bytes decode to as a payload. Most block payloads in this format are not plaintext; they are structured data of some kind, with their own internal validity rules (field types, lengths, embedded checksums, expected formats). A successful forgery that is also meaningful — readable, parseable, or exploitable by whatever consumes the payload — requires separately satisfying that structural validity, on top of the hash collision and on top of the ROM-address constraints of Section C.

These three requirements — valid ROM addressing, hash-target collision, and application-level semantic validity — are structurally unrelated to one another. Success against one provides no leverage toward the others. A meaningful forgery is not merely improbable; it requires simultaneously winning three independent searches, none of which share a common shortcut.

3. Scalable Architecture: Branch and Trunk Chains

The ZOSCII Tamperproof Blockchain introduces a branching architecture for enhanced scalability:

Genesis Block (64KB, byte[0]=0) ↓ Trunk Block 1 ↓ Trunk Block 2 ├─→ Branch A Block 1 │ ↓ │ Branch A Block 2 │ ├─→ Branch B Block 1 │ ↓ │ Branch B Block 2 ↓ Trunk Block 3

A. Trunk Chain

The main chain that serves as the authoritative timeline. All branches anchor to a block on the trunk, and consensus reconciliation selects the canonical trunk during synchronization.

B. Branch Chains

Chains that split from any existing block — typically a trunk block. A branch block is structurally identical to a normal block (block_type = Normal); what distinguishes it is two fields in its raw header. Each branch:

  • Sets is_branch = 1 in its raw header
  • Records its parent trunk's chain ID in the trunk_id field
  • Builds its Rolling ROM the same way as any block — walking backwards via prev_block_id, which naturally passes from the branch's own history into the trunk's history once it reaches the branch's first block
  • Can be verified independently or together with the trunk, by walking back through it
Use Cases for Branches:
  • Wallets: Each wallet has its own transaction chain
  • Departments: Company trunk with departmental branches
  • Projects: Main project trunk with sub-project branches
  • Time Periods: Financial year trunk with quarterly branches

4. Technical Specifications

Genesis Block

  • Size: 64KB (65,536 bytes) exactly — this is how every tool identifies it on disk
  • Byte 0: block type marker, value 0
  • Bytes 1–65535: high-entropy data, XOR-blended from 1–3 source files
  • Coverage: the entropy region should contain a good spread of all 256 byte values
  • Purpose: immutable root of trust

Rolling ROM

  • Size: 64KB (65,536 bytes)
  • Construction: samples from up to 64 previous blocks (1KB each, from their complete serialized bytes), reached by walking backwards via prev_block_id
  • Fill order: walked-back block history, then a Truncation block's stored ROM snapshot if one is reached, otherwise the Genesis Block
  • Purpose: dynamic encoding/decoding table, specific to each individual block

Block Encoding

  • Input: 1 byte of data
  • Output: 2-byte pointer (address in Rolling ROM)
  • Selection: random from all available addresses holding that byte value
  • Decoding: single O(1) lookup operation
  • Padding: minimum payload size 512 bytes; padding bytes are not zero-filled in the C# reference (xorshift32-derived) and zero-filled in the C tools — padding content has no security role, only the declared payload length is meaningful

Block Types

  • Genesis (0) — The 64KB root-of-trust block
  • Normal (1) — Standard data block; also used for branch blocks, which are distinguished by the is_branch/trunk_id header fields rather than a distinct type
  • Checkpoint (2) — Documented anchor point, payload is a label string
  • Truncation (3) — Written by the truncation operation; payload is a raw, unencoded 64KB rolling ROM snapshot. Chain walks stop here; the chain above is self-contained without the original genesis block.
  • Finalise (4) — Soft close marker, payload is a label string; signals no further blocks should be appended
  • Bridge (5) — Reserved for future cross-chain linking

Block Filename and Layout

  • Filename: <blockID>.ztb — no chain ID or index is encoded in the filename; chain membership and position live entirely in the block's own header fields
  • Layout (non-genesis, non-truncation): 111-byte raw, unencoded header (block_type, block_version, is_branch, trunk_id, block_id, prev_block_id) followed by a ZOSCII-encoded section (hash_type, hash, prev_hash, payload_len, padded_len, payload)
  • Layout (truncation): the same 111-byte raw header, followed directly by a raw, unencoded 65,536-byte rolling ROM snapshot

Security Guarantee

  • Tamper Probability: ~10^-152,900 (combinatorially infeasible)
  • Quantum Proof: Yes (no mathematical problems to attack)
  • Information-Theoretic: Yes (security independent of computation)

7. Security, Confidentiality, and Consensus

A. Quantum Proof

The ZOSCII Tamperproof Blockchain is Quantum Proof by Design. Its security is rooted in the combinatorial explosion of the pointer-encoding keyspace and Information-Theoretic Security (due to the high-entropy ROMs and non-reproducible encoding), which are immune to known quantum algorithms like Shor's.

B. Flexible Payload Security

The integrity of the chain structure is secured separately from the confidentiality of the data payload. The data packet area of a block can contain:

  • Visible Data: Fully transparent records for public verification.
  • ITS/ZOSCII Encoded Data: Data that is secure from eavesdroppers but quickly decoded by authorized ZOSCII systems.
  • External Pointers/URLs: Metadata linking to data stored off-chain.

C. Consensus (Proof of Integrity)

A consensus mechanism (which can be a light-weight Proof-of-Work or Proof-of-Stake to control block rate) layered on top of this format would primarily validate Integrity and Validity:

  • The new block's structure correctly utilizes the rolling ROM derived from its lineage
  • The new block is fully traceable back to the Genesis Block (or an active Truncation block) and, if applicable, the branch's anchor block

D. Application: The 'Proof of Something' Chain

The architecture is ideal for a Generic 'Proof of Something' Chain. A real-world document (e.g., a university degree) can contain an identifier that points directly to a block. Verification simply requires checking the block against the chain's integrity rules, providing publicly verifiable and transparent proof of a claim's integrity since the moment it was recorded.
